This project is a static site built with [aphid](https://github.com/LHelge/aphid), a Rust
static site generator that builds a blog, a wiki, and standalone pages from Markdown files
with `[[wiki-link]]` cross-referencing.

**Keeping these instructions current**: this file and the accompanying skill files are
generated by `aphid agent <tool>` and embedded in the aphid binary. After upgrading aphid,
re-run the same `aphid agent` command — the skill files will be refreshed so they describe
features added by the new release. This main file is preserved on re-run so you can extend
it with project-specific guidance; delete it first if you want a clean regeneration.

# Repository layout

- `aphid.toml` — site configuration (title, base URL, theme directory, authors, socials).
- `content/blog/` — dated blog posts.
- `content/wiki/` — reference / wiki pages.
- `content/pages/` — standalone pages (About, Contact, etc.).
- `content/home.md` — optional home page content.
- `theme/` (or whatever `theme_dir` points at) — Tera templates and static assets.
- `static/` — user static files; copied as-is into the build output.
- `dist/` — generated build output (gitignored).

# Common commands

- `aphid serve` — start the dev server on port 3000 with file watching and live reload. `--port` overrides the port.
- `aphid build` — render the site into `dist/`. `--output <dir>` overrides the directory.
- `aphid new <name>` — scaffold a complete site in a new directory.
- `aphid init [path]` — scaffold a site in the current (or given) directory.
- `aphid blog new "Title"` — scaffold a new blog post (filename gets today's date prefix).
- `aphid wiki new "Title"` — scaffold a new wiki page.
- `aphid page new "Title"` — scaffold a new standalone page.
- `aphid agent <tool>` — regenerate the AI-agent instruction files. `<tool>` is `claude`, `codex`, `copilot`, etc. Pass no tool for a generic `AGENTS.md`. Run this after upgrading aphid to refresh the skill files with any new features.

The global `--config <path>` flag (default `aphid.toml`) overrides the config file location on every subcommand.

# Wiki-link conventions for this project

A few writing rules learned the hard way — apply these when authoring or editing any
content under `content/`:

- **Use natural prose form for multi-word slugs.** Aphid slugifies the target text inside
  `[[ ... ]]` (lowercase, spaces → hyphens, hyphens preserved), so write
  `[[VCM requirements]]`, `[[EV-CAN]]`, `[[HV system]]`, `[[Aphid project]]` instead of
  `[[vcm-requirements]]`, `[[ev-can]]`, etc. The rendered href and display text are
  identical either way — display always comes from the target page's `title` frontmatter —
  but the natural form reads better in source. Single-word slugs (`[[motor]]`, `[[vcm]]`)
  are fine as-is.
- **Never use `[[slug|display]]` inside a markdown table cell.** The `|` collides with the
  table-cell separator and the link won't render. In tables, always use the bare form
  `[[slug]]` (or `[[Natural form]]`) and let the target page's title supply the column
  text. If the title doesn't fit the column, rename the page rather than reintroducing
  `|display`.
- **Outside tables, drop `|display` when the target page's title already reads naturally
  in the sentence.** Keep `|display` only when the title is too verbose inline (e.g.
  `[[pdm|PDM]]` because the title is "Nissan Leaf PDM (Power Delivery Module)"), when a
  contextual phrase is genuinely useful (`[[pdm#dcdc|DC/DC built into the PDM]]`,
  `[[batteries|battery pack]]`), or to keep parallelism inside a list of short references.
- **Wiki slug style.** Keep slugs (= wiki filenames) short and contextual — e.g. `motor.md`
  rather than `nissan-leaf-em57-motor.md`. The descriptive form lives in the page's `title`
  frontmatter, and the rest of the content makes the referent obvious.

# Hero image sizing

Blog post `image:` frontmatter feeds both the in-post hero (`.article-hero`) and the
blog-index featured/row thumbnails — one asset, three crops. Target the largest use case:

- **Export at 1920 × 960** (2:1 aspect ratio, WebP, quality ~80, aim under ~250 KB).
- Display max is 960 × 480 on the no-TOC post layout (`.page-narrow`) and 720 × 480 on the
  TOC layout (`.page-grid-post`); 1920 px wide covers 2× retina at 960 px display.
- The hero uses `object-fit: cover` with `max-height: 480 px`, so anything taller than 2:1
  gets cropped — keep the subject centred.
- **WebP is preferred** for photos and hero images. Keep PNG for diagrams with hard edges
  or fine text (`vcm_pinout.png`, `inverter_pinout.png`) and for `static/logo.png` (used as
  the favicon source). If you're worried about a specific social-share scraper for the
  site-wide `social_image`, prefer PNG/JPG there; everywhere else WebP is fine.
